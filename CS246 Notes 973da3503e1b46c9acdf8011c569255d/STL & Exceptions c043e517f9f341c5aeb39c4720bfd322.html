STL & Exceptions

C++ Templates

  • A feature of C++ programming language
  • Allows functions and classes to operate with generic types
  • Allows functions and classes to work on different data types without re-writing the code

Why we need templates?

Scene: Assume that our client wants us to implement ADT Stack using dynamic arrays, however, he wants the Stack to support int, char, std::string, so what should we do? Based on what we have learned so far, we have no choice but to implement 3 different Stacks, one supports int, one supports char, and one support std::string. So why bother writing the same code three times? Therefore, we need templates to simplify this process.

Syntax: template<typename T> (no semicolon ;)

Interpretation:

  • Telling the compiler that we have some data type T.
  • We replace T with specific data type when calling functions or classes.
  • We can avoid writing the same code for supporting different data types.

Example: Write a function that finds the maximum of two integers or two chars.

Old Method:

using namespace std;
// using function over loading
int findMax(int a, int b) {
	return a >= b ? a : b;
}

int findMax(char a, char b) {
	return a >= b ? a : b;
}

int main() {
	cout << findMax(3,5) << endl;
	cout << findMax('a', 'B') << endl;
}

New Method (using template):

template<typemname T>
T findMax(T a, T b) {
	return a >= b ? a : b;
}

int main() {
	cout << findMax(3,5) << endl;
	cout << findMax('a', 'B') << endl;
}

Then, for the Stack ADT example, we can implement it as following:

template <typename T>
class Stack {
	int size = 0, cap = 10;
	T* arr;
public:
	explicit Stack();
	~Stack();

	void push(const T&);
	T pop();
	int size() const;
}

Stack::Stack(): // However, this would cause error...
	arr{new T[cap]} { } // see below for reason

Reason:

  • We must let the compiler know the specific type when compiling.
  • So instead of writing Stack, we must write Stack<T> where T will be replaced by a specific data type when create Stack objects
  • And also, remember that template statement is single-use! We have to redefine it every time a function uses it! Thus, we have to add template <typename T> before the function definition.
  • *Remember we did not add semicolon after template <typename T>, this means that this is not a complete command, and it has to be bound with something.

Correct implementation:

template <typename T>
class Stack {
	int size = 0, cap = 10;
	T* arr;
public:
	explicit Stack();
	~Stack();

	void push(const T&);
	T pop();
	int size() const;
}

template <typename T>
Stack<T>::Stack():
	arr{new T[cap]} { } // now this can compile

Other functions in Stack ADT:

template<typename T>
Stack<T>::~Stack() {delete [] arr};

template<typename T>
void Stack<T>::push(const T& item) {
	if (size == cap) {
		cap *= 2;
		T* newArr = new T[cap];
		for (int i = 0; i < size; i++) {
			newArr[i] = arr[i];
		}
		delete [] arr;
		arr = newArr;
	}
	arr[size++] = item;
}

template<typename T>
T Stack<T>::pop() {return arr[--size]};

template<typename T>
int Stack<T>::length() const { return size; }

How to use it?

int main() {
	Stack<int> s1;
	Stack<char> s2;
	Stack<std::string> s3;
}
  • Now our code supports three data types
  • T can be any data type that compiler recognizes, even user-defined structure or class is recognized

IMPORTANT:

  • Template classes and member function definitions must be in the header file!!!
  • Template declarations definitions must be in the same file! (No way to link templates from different files!)

Standard Template Library (STL)

  • A set of template classes to provide common programming data structures and functions. (C++ built-in libraries)
  • A library of container classes, algorithms and iterators.

Four components in STL:

  • Algorithms
  • Containers → We mainly focused on this one in this lecture
  • Functions
  • Iterators

Containers:

  • Sequence containers: vector, list, deque, arrays, forward_list
  • Container adaptors: queue, stack, priority_queue
  • Associative containers: map, set, multiset, multimap
  • Unordered associative containers: unordered_map, unordered_set

Vector

  • A sequence container representing array that can change its size.
  • Uses dynamic array to store elements (provides random access)
  • May allocate extra storage for possible growth (might be doubling strategy).

How to use it?

#include <vector> allows us to use the template class Vector.

Since vector is a template class, we need to tell compiler the specific data type when we create the object.

e.g.: std::vector<int> v;

Vector documentation website: https://cplusplus.com/reference/vector/vector/

Some commonly used functions:

  • emplace_back( ... ) : insert element at the end of the array.
  • pop_back() : removes and returns the last element in array.
  • size() : returns the number of elements in array.
  • clear() : removes all elements in vector.

    Note: clear() only erases the elements, and that if the elements themselves are pointers, the pointed-to-memory is not touched in any way. Managing the pointers is the user’s responsibility. Remember to deallocate heap memories the pointers point to before removing them from the vector!

Two Methods to Access Elements in Vector:

Method 1: operator[]

for (unsigned int i = 0; i < v.size(); i++) {
	std::cout << v[i] << std::endl;
}

Method 2: at( int )

for (unsigned int i = 0; i < v.size(); i++) {
	std::cout << v.at(i) << std::endl;
}

Difference between those 2 methods:

  • operator[] does not check boundaries and may cause undefined behaviors.

  • at() checks boundaries
  • It throws an exception as soon as the index i is out of bound.

Exceptions

Let’s take a closer look of the error message we just saw

  • Program threw an exception.
  • Exceptions provide a way to transfer control from one part of a program to another.
  • Since our code could not handle exceptions, program is aborted.

Basic control flow:

  • Conditions (if, else if, else)
  • Iterations (while-loops, for-loops)
  • Return / function calls
  • Exceptions

How do we handle exceptions?

  • If program throws an exception (basically an error)
  • we need to catch the exception, otherwise program terminates.

How do we catch exceptions?

Example: catch the exception thrown by read fail

Instead of perform the operations directly, we have to “try” to perform the operations.

#include <iostream>

int main() {
	int num;
	
	for ( ; ; ) {
		// try to read integers
		try {
			std::cin >> num;
			// throw exception if read fails
			if (std::cin.fail()) {
				// suppose we throw an interger
				int num = -1;
				throw num;
 		}
	}
}

If read successes, we just print the integer. If read integer fails, we throw the exception. How do we catch the exception?

#include <iostream>

int main() {
	int num;
	
	for ( ; ; ) {
		// try to read integers
		try {
			std::cin >> num;
			// throw exception if read fails
			if (std::cin.fail()) {
				// suppose we throw an interger
				int num = -1;
				throw num;
 		} catch (int& error) {
			// int type exception will be caught
			std::cin.clear();
			std::cin.ignore();
			if (std::cin.eof()) break;
	}
}

An easier approach of handling read fail problem:

#include <iostream>

int main() {
	int num;
	
int main() {
	int num;

	// read will throw exceptions if fail
	std::cin.exceptions(std::ios::failbit);

	for ( ; ; ) {
		try {
			std::cin >> num;
 		} catch (std::ios::failure&) {
			// int type exception will be caught
			std::cin.clear();
			std::cin.ignore();
			if (std::cin.eof()) break;
	}
}
  • Explicitly let std::cin throws an exception if read fails
  • Catch the build-in error type std::ios::failure
  • If you are not familiar with C++ build-in exceptions, you can always write customized exception.

Exception Objects

//.h file
#include <iostream>

class BaseException {
public:
	virtual void print() const {
		std::cout << "exception from base class" << std::endl;
	}
};

class DerivedException : public BaseException {
public:
	void print() const override{
		std::cout << "exception from derived class" << std::endl;
	}
};

//.cc file
int main() {
	DerivedException de;
	
	try {
		throw de;
	}
	catch (BaseException be) {
		be.print();
	}
	catch (DerivedException de) {
		de.print();
	}
	std::cout << "finish executing" << std::endl;
}
  • The second catch block will never be executed
  • By definition of inheritance, we know that DerivedException is BaseException.
  • Code compiles but gives us warning, because this is not how C++ is designed.

IMPORTANT:

Always remember to catch derived classes first, then base class!

Correct implementation:

//.cc file
int main() {
	DerivedException de;
	
	try {
		throw de;
	}
	catch (DerivedException de) {
		de.print();
	}
	catch (BaseException be) {
		be.print();
	}
	std::cout << "finish executing" << std::endl;
}

Recall object slicing:

Object slicing occurs at line 23, thus line 24 executes method in base class.

Solution:

 //.cc file
int main() {
	DerivedException de;
	
	try {
		throw de;
	}
	catch (BaseException& be) { // catch by reference
		be.print();
	}
	std::cout << "finish executing" << std::endl;
}

Now, after we throw by reference, we are using dynamic dispatch and there is no object slicing.

Stack unwinding:

Example:

void goo() {
	std::string str = "error";
	throw str;
}

void foo(int num) {
	if (num % 2 == 0) {
		goo();
	}
	std::cout << "foo is finished" << std::endl;
}

int main() {
	int num = 10;
	try {
		foo(num);
		std::cout << "foo finishes executing" << std::endl;
	}  catch (std::string&) {
		std::cout << "exception from foo is caught" << std::endl;
	}
	std::cout << "program terminates" << std::endl;
}

Results:

// when num = 9 (odd number):
// output:
>>> foo is finished
>>> foo finishes executing
>>> program terminates


// when num = 10 (even number):
// output:
>>> exception from foo is caught
>>> program terminates

If we change the code into the following:

void goo() {
	std::string str = "error";
	throw str;
}

void foo(int num) {
	if (num % 2 == 0) {
		try {
			goo();
		} catch {std::string&) {
			std::cout << "exception from goo is caught" << std::endl;
			throw; // re-throw the same exception object
		}
	}
	std::cout << "foo is finished" << std::endl;
}

int main() {
	int num = 10;
	try {
		foo(num);
		std::cout << "foo finishes executing" << std::endl;
	}  catch (std::string&) {
		std::cout << "exception from foo is caught" << std::endl;
	}
	std::cout << "program terminates" << std::endl;
}

Now, the result becomes:

// when num = 10 (even number):
// output:
>>> exception from goo is caught
>>> exception from foo is caught
>>> program terminates

Stack unwinding: part of the function commands in the stack frame are not executed because the program terminates early due to an exception thrown.

Map

  • An associative container that store elements in a combination of key-value pair.
  • Keys are used to sort.
  • Map is always sorted in an ascending order.

Map documentation website:

https://cplusplus.com/reference/map/map/

Syntax: #include <map> allows us to use the template class map.

#include <iostream>
#include <map>

int main() {
	// each element in map is std::pair
	std::map<int, char> m; // the first argument is the type of keys
												 // the second argument is the type of values
	// insert elements
	m.emplace(3, 'f'); // 1st arg is the key, 2nd arg is the value
	m.emplace(1, 'b');
	m.emplace(2, 'z');
	m[5] = 'k'; // another method to assign a value to a key
	m[3] = 'g';

	for (std::pair<int,char> elem : m) {
		std::cout << elem.first << ' ' << elem.second << std::endl;
	}
}
// elem.first retrieves the key in the pair.
// elem.second retrieves the value in the pair.

Map command: find

Finding elements in map:

std::map<int, char>::iterator it = m.find(3);
if (it == m.end()) { // item is not found
	std::cout << "element not found" << std::endl;
} else {
	std::cout << "element is found, value: " << it->second << std::endl;
}

// m.find(key) find the iterator with key = 3.
// Dereferencing a map iterator (within the range) will get a std::pair.
// If no pair with key = 3 is found, then it == m.end() (end iterator).

Command: m.find(key)

  • Finding elements by searching the key
  • find method returns an iterator type.

Map command: count

if (m.count(5) == 0) { // if key = 5 is not found
	std::cout << "element is not found" << std::endl;
} else { // if key = 5 if found
	std::cout << "element is found with value: " << m[5] << std::endl;
}

Command: m.count(key)

  • Counts the number of existing keys matching with the argument key.
  • Return type is int (only return 0 or 1, since keys are all unique)

Compilation dependencies

We use include guards to avoid multiple definitions in header files. However, there is one risk condition using include guards. It occurs if there exists an including cycle.

Example:

// cell.h
#ifndef __CELL_H__
#define __CELL_H__

#include "list.h"

struct Cell {
	List* l;
};

#endif

// list.h
#ifndef __LIST_H__
#define __LIST_H__

#include "node.h"

struct List {
	Node* root;
};

#endif
// node.h
#ifndef __NODE_H__
#define __NODE_H__

#include <vector>
#include "cell.h"

struct Node {
	std::vector<Cell> v;
};

#endif
// circle.cc (main file)
#include "cell.h"
#include "node.h"
#include "list.h"

int main() {

}

Will this code compile?

Answer: No.

Error message:

Reason:

Node is defined first. Node contains a vector of Cell but Cell is not defined。 Circular including occurs.

If we must define these three structures in this way, how should we fix it?

Answer: Instead of including the corresponding header files, we just add a forward declaration (opaque structures).

Example:

// cell.h
#ifndef __CELL_H__
#define __CELL_H__

class List; // using forward declarating instead
					  // of including the header file

struct Cell {
	List* l;
};

#endif
  • We use forward declarations when deal with pointers/references, since pointers are always 8 bytes.
  • Function declarations only needs forward declarations, however, must include corresponding header files when implementing functions.
  • In other cases, we must include header files.

Classical sample cases:

// a.h
class A {
 ...
};

Case 1:

// b.h
#include "a.h" // include header file

class B {
	A a;
};

When class B contains a field of type A (or in cases that B needs to know exactly how many bytes a class A object takes to determine the size of class B objects), we must include A’s header file.

Case 2:

// b.h
class A; // forward declaration

class B {
	A& a1;
	A* a2;
};

When class B contains fields of class A reference or pointer to class A objects, (or in cases that B does not need to know exactly how many bytes a class A object takes), we can just simply use forward declarations.

Case 3:

// b.h
class A; // forward declaration

class B {
	void foo(A a);
};

When class B contains function declarations with class A object as parameters, we can simply use forward declaration, since parameter types in function declarations does not take memories.

Case 4:

// b.h
#include "a.h" // include header file

class B {
	void foo(A* a) { // function definition
		...
	}
};

When class B contains function definitions with class A objects as parameters, it depends:

  1. if it’s passing by value, then we must include the corresponding header file!
  1. if it’s passing by reference or by pointer, if we access memories through the pointer or the reference, then we must include the corresponding header file! If we don’t access the corresponding memories, then we can simply use forward declaration.
